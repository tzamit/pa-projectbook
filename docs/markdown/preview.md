# מסמך אפיון - מערכת PaymentsAid
## סקירה תפעולית ותהליכית

---

## 1. מהות המערכת

מערכת PaymentsAid היא מערכת תשלומי סיוע לאוכלוסיות מיוחדות במשרד הרווחה (מוסא). המערכת מנהלת תשלומים חודשיים שוטפים ורטרואקטיביים עבור חמש תוכניות סיוע:

| סמל | תוכנית | אוכלוסייה | סדר גודל |
|------|---------|-----------|-----------|
| 5 | דמי ליווי | עיוורים | ~9,000 מוטבים |
| 6 | דמי תקשורת | חירשים | ~800 מוטבים |
| 7 | מונשמים | מונשמים | קטן |
| 8 | כלבי נחייה | עיוורים עם כלב נחייה | קטן |
| 9 | החזקת טלפון | חירשים | ~20,000 מוטבים |

**תקציב שנתי:** סדר גודל של עשרות מיליוני ש"ח (דמי ליווי לבדם כ-22 מיליון ש"ח).

---

## 2. ישויות מרכזיות

המערכת מנהלת שלוש ישויות עסקיות מרכזיות:

### 2.1 מוטב (Beneficiary)
- לכל מוטב מוצמד **קוד הקצאת בנק** ייחודי במרכבה
- לאדם יכולים להיות מספר חשבונות בנק, כל אחד עם קוד הקצאה נפרד
- כל משרד מחזיק קוד הקצאת בנק משלו
- מוטבים חדשים נקלטים במערכות ומוקמים במרכבה עם פרטי הבנק שלהם

### 2.2 בקשת תמיכה (Support Application)
- מקבילה ל"התחייבות" — מנהלת את **ההקצאה השנתית** של התקציב למוטב
- בינואר נפתחות בקשות תמיכה לכל המוטבים (סכום שנתי = תעריף חודשי × 12)
- במהלך השנה מתעדכנות בקשות רק כשיש שינוי: תעריף רטרואקטיבי, תשלום חריג, פטירה וכד'
- סדר גודל: ~9,000 בקשות בינואר, עשרות בודדות בשאר החודשים

### 2.3 דרישת תשלום (Payment Request)
- מנהלת את **התשלום בפועל** — כמה לשלם למוטב בחודש נתון
- נוצרת אוטומטית בעיבוד החודשי
- מנוהלת ברמת **חודש דיווח** וב**חודש עיבוד** (ייבוא)
- כשחודש דיווח ≠ חודש עיבוד — מדובר בתשלום רטרואקטיבי

## 3. תהליך העיבוד החודשי

העיבוד החודשי הוא לב המערכת. מתבצע כל חודש לכל תוכנית סיוע בנפרד.

### 3.1 שלבי העיבוד

```
┌─────────────────────────────────────────────────────────────────┐
│  1. ביצוע עיבוד חודשי                                          │
│     → אכלוס טבלאות בנתוני דרישות תשלום ובקשות תמיכה            │
│                                                                 │
│  2. בדיקת מוטבים (אופציונלי, בכל שלב)                          │
│     → זיהוי מוטבים בעייתיים שהקמתם במרכבה לא הושלמה            │
│                                                                 │
│  3. העברת בקשות תמיכה למרכבה                                    │
│     → שליחה דרך מערכת התווך → קבלת איזון חוזר                   │
│     → בדיקת בקשות שגויות וטיפול                                 │
│                                                                 │
│  4. אישור חשבות + פריסת אבני דרך                                │
│     → תהליך אישור ע"י גורם בחשבות במרכבה                       │
│                                                                 │
│  5. העברת דרישות תשלום למרכבה                                   │
│     → שליחה דרך מערכת התווך → קבלת איזון חוזר                   │
│     → בדיקת דרישות שגויות וטיפול                                │
│                                                                 │
│  6. קידום חודש                                                  │
│     → העברת המערכת לחודש הבא (פעולה בלתי הפיכה)                │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 בדיקות במהלך העיבוד

**בדיקת מוטבים** — מזהה מוטבים שאמורים לקבל תשלום אך הקמתם במרכבה לא הושלמה:
- חשבון בנק שגוי
- תשלום שחזר (חשבון נסגר / בעיה בהעברה)
- מוטב שנשלח כ"חדש" אך כבר קיים במרכבה (צריך לשלוח כ"עדכון")
- חסרים פרטי בנק תקינים

**בדיקות זכאות** — מתבצעות כחלק מהעיבוד:
- בדיקת השמה במסגרת פנימית (מסר) — מי שמושם במסגרת לא זכאי לדמי ליווי
- בדיקת תאריכי פטירה מול מרשם אוכלוסין (מתעדכן מידי לילה ממשרד הפנים)
- בדיקת סטטוס תעסוקה (רלוונטי לחירשים — תעריף שונה לעובד/לא עובד)

### 3.3 לוחות זמנים

- התשלום מתבצע ב-20 לכל חודש — **ללא גמישות**
- העיבוד אמור להסתיים תוך שבוע בערך
- לאחר סיום — קידום לחודש הבא (אין חזרה)

### 3.4 בעיית ביצועים (צוואר בקבוק נוכחי)

העיבוד החודשי סובל מבעיות ביצועים חמורות:
- **Timeout** חוזרים — העיבוד נופל אחרי ~10 דקות
- הבעיה חמורה במיוחד ב**דמי ליווי** ו**דמי תקשורת**
- **החזקת טלפון** — 20,000 מוטבים × 12 תשלומים בבת אחת (בינואר) — עומס קריטי
- הפתרון הנוכחי: הרצה חוזרת עד שמצליח — **לא בר-קיימא**
- הסיבה: נפח נתונים גדל עם הזמן, המערכת לא תוכננה לכמויות הנוכחיות

## 4. מערכת התווך (Middleware)

מערכת פנימית שנבנתה באגף, משמשת כגשר בין המערכות המשלמות למרכבה:

```
PaymentsAid  ──→  מערכת התווך  ──→  כספת משותפת  ──→  מרכבה
                                  ←── איזונים חוזרים ←──
```

- שולחת קבצים לכספת משותפת עם מרכבה
- מקבלת איזונים חוזרים ומעדכנת את המערכות
- בקבלת איזון חוזר — נשלח מייל לעובדת ולמנהלת

### קודים חדש/עדכון
בשליחה למרכבה חובה לציין האם המוטב **חדש** או **עדכון**:
- אם נשלח כ"חדש" והמוטב כבר קיים — מרכבה מחזירה שגיאה (לא מבצעת עדכון אוטומטי)
- יש לשלוח שוב עם קוד "עדכון"

## 5. תשלומים רטרואקטיביים

### 5.1 מתי נדרש תשלום רטרואקטיבי?
- **הכרה מאוחרת** — אדם שהושלם תהליך ההכרה שלו היום אך זכאי חצי שנה אחורה
- **שינוי סטטוס תעסוקה** — חירש שלא הגיש אישור עבודה בזמן, ירד לתעריף נמוך, ואח"כ הגיש
- **עדכון תעריף רטרואקטיבי** — נדיר, לא אמור לקרות בשגרה

### 5.2 מנגנון החישוב
- המערכת יוצרת **מינוס** על דרישות התשלום הישנות
- יוצרת דרישות חדשות עם התעריף המעודכן
- המוטב מקבל בפועל רק את **ההפרש**
- כל תשלום מנוהל ברמת חודש דיווח + חודש עיבוד

### 5.3 טבלאות נתונים
- **טבלת ריכוז (RSAIP)** — פירוט חודשי מדויק: כמה קיבל בכל חודש
- **טבלת חשבות** — דרישת תשלום כפי שעוברת למרכבה: סכום כולל אחד למוטב לחודש עיבוד (שורה אחת = סכום נטו = שוטף + רטרו)

## 6. תעריפים

### 6.1 מבנה התעריפים

| תוכנית | תעריף (סדר גודל) | הערות |
|---------|-------------------|-------|
| דמי ליווי (עיוורים) | ~2,400 ש"ח/חודש | אחיד לכולם, צמוד לקצבת ניידות של ביטוח לאומי |
| דמי תקשורת (חירשים) | ~500 ש"ח/חודש | תעריף שונה לעובד/לא עובד |
| החזקת טלפון | נמוך | תשלום שנתי, אך נוצרות 12 רשומות חודשיות |

### 6.2 עדכון תעריפים
- מתבצע דרך מסך ייעודי במערכת
- עיוורים — צמוד לעדכוני ביטוח לאומי (קצבת ניידות)
- חירשים — מחושב במשרד עצמו
- העדכון לא בהכרח בינואר (מחכים למדד)
- עדכון רטרואקטיבי לכלל האוכלוסייה — לא נתמך אוטומטית (נדרש פיתוח ייעודי)

## 8. דוחות עיקריים

| דוח | שימוש | תדירות |
|-----|-------|--------|
| מקבלי תשלום בבנק | בקרה מול חשבות — התאמת סכומים 100% לפני שחרור כספים | חודשי |
| מקבלי תשלום בהמחאות | הדפסת שוברים לתשלום במזומן | חודשי |
| דוח סטטיסטי | תמונה כוללת — התפלגות מוטבים, סכומים, קשישים | לפי צורך |
| שאילתות | מענה לפניות מוטבים — "כמה קיבלתי?" | שוטף |

### תהליך בקרת חשבות
1. העובדת מפיקה דוח מקבלי תשלום בבנק
2. מייצאת לאקסל ושולחת לחשבות
3. חשבות מבצעת התאמה מול מרכבה
4. רק לאחר התאמה מלאה — הכסף משוחרר לתשלום

## 11. מערכות משיקות ועתיד

### מערכת תשלומים חדשה (בפיתוח)
- נבנית ע"י חברת מטריקס
- מיועדת לאחד את כל מערכות התשלומים של המשרד (אנשים פרטיים + מסגרות)
- המערכות ייכנסו בהדרגה
- מט"ס (מערכת ותיקה, בת 54 שנה) — ראשונה להיכנס

### סל תקשורת + הטבות חברויות לעיוורים
- מערכת חדשה שכבר הוקמה
- סל תקשורת — תשלום כנגד הצגת קבלות (החזר חלקי)
- הטבות חברויות — מקלות נחייה, ציוד וכד'
- כיום המוטב רוכש ומגיש קבלה בדיעבד — המערכת החדשה מאפשרת קליטת חשבוניות

### תשלומים היסטוריים
- נתונים מלפני 2008 — מתקופת מערכת מט"ס הישנה
- הרשויות המקומיות היו מדווחות ומזמינות תשלום
- נשמרים במערכת לצורך היסטוריה בלבד, לא פעילים
